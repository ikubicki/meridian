# Findings: phpBB Installer Bootstrap & Multi-Container Patterns

**Source category**: `codebase-installer`
**Research question**: How does the phpBB installer bootstrap differently from the main forum? How can `web/install/api.php` bootstrap without triggering the redirect-to-installer guard?

---

## 1. The Redirect Guard — `PHPBB_INSTALLED`

### Where the constant is defined

`src/phpbb/common/config/config.php:14`
```php
@define('PHPBB_INSTALLED', true);
```
This file is auto-generated by the installer task
`src/phpbb/forums/install/module/install_filesystem/task/create_config_file.php:193`:
```php
$config_content .= "\n@define('PHPBB_INSTALLED', true);\n";
```
Until the installer runs, `config.php` either does not exist or does not contain this definition.

### Where the redirect is triggered

`src/phpbb/common/common.php:30`
```php
\phpbb\install\checkInstallation($phpbb_root_path);
```
This is called **unconditionally** near the top of `common.php`, before the DI container is built. Every front-controller that does `require 'common.php'` (i.e. `web/app.php`, `web/index.php`, etc.) will redirect to the installer if `PHPBB_INSTALLED` is not defined.

### What `checkInstallation()` does

`src/phpbb/install/redirect.php:22-27`
```php
function checkInstallation(string $phpbb_root_path): void
{
    if (!defined('PHPBB_INSTALLED'))
    {
        redirectToInstaller($phpbb_root_path);
    }
}
```
The check is purely a constant test. `redirectToInstaller()` builds its own redirect URL and calls `header()/exit` — no exceptions, no recoverable path.

### `install_helper::is_phpbb_installed()` — slightly richer logic

`src/phpbb/forums/install/helper/install_helper.php:48-70`

The installer's own helper uses a three-tier check:
1. `defined('PHPBB_INSTALLED')` → return `true`
2. `getenv('PHPBB_INSTALLED') !== false` (env var set without a constant) → return `false` (env var presence means "not installed via env")
3. File heuristic: `config.php` exists **and** no `cache/install_lock` → `true`

This is used only inside the installer's own controllers/navigation, not in `common.php`.

## 2. How the Installer Bootstraps — Completely Separately

### Entry point

`web/install.php` (lines 1-17):
```php
define('PHPBB_ROOT_PATH', __DIR__ . '/../');
require('../src/phpbb/install/app.php');
```

No `common.php` is ever included. The installer has its own fully independent boot path.

### `src/phpbb/install/app.php` — defines `IN_INSTALL`

`src/phpbb/install/app.php:17`
```php
define('IN_INSTALL', true);
```
Then it includes `src/phpbb/install/startup.php`.

### `startup.php` guard

`src/phpbb/install/startup.php:14-17`
```php
if (!defined('IN_INSTALL'))
{
    exit;
}
```
All installer support files share this guard — they cannot be included from outside the installer context.

### Separate DI container built in `startup.php`

`src/phpbb/install/startup.php` (near end of file):
```php
$phpbb_installer_container_builder = new \phpbb\di\container_builder($phpbb_root_path);
$phpbb_installer_container_builder
    ->with_environment('installer')
    ->without_extensions();

$config_path = $phpbb_root_path . 'src/phpbb/common/config';

$phpbb_installer_container = $phpbb_installer_container_builder
    ->with_config_path($config_path)
    ->with_custom_parameters(['cache.driver.class' => 'phpbb\cache\driver\file'])
    ->get_container();
```

Key differences vs the forum container:
- `->with_environment('installer')` → selects `src/phpbb/common/config/installer/` YAML tree instead of `default/`
- `->without_extensions()` → phpBB extensions are not loaded
- No `->with_config($phpbb_config_php_file)` → the `config.php` variables (DB credentials, etc.) are **not** injected; the `dbal.conn` service is not wired

### Installer DI config tree

`src/phpbb/common/config/installer/config.yml`:
```yaml
imports:
    - { resource: ../default/config.yml }
```

`src/phpbb/common/config/installer/container/services.yml` imports only:
- `services_installer.yml` (installer-specific services)
- `services_event.yml`, `services_filesystem.yml`, `services_http.yml`
- `services_language.yml`, `services_php.yml`, `services_routing.yml`, `services_twig.yml`

**Conspicuously absent**: `services_database.yml`, `services_user.yml`, `services_session.yml`, and the full forum service tree. The installer container is intentionally lightweight.

`src/phpbb/common/config/installer/container/parameters.yml`:
```yaml
imports:
    - { resource: ../../default/container/parameters.yml }
parameters:
    installer.create_config_file.options: []
```

### Class loading in installer mode

`startup.php` registers three separate autoloaders:
1. `phpbb\` → `install/update/new/phpbb/` (for update scenarios)
2. `phpbb\` → `src/phpbb/forums/` (main phpBB classes)
3. `phpbb\convert\` → `src/phpbb/install/convert/`

## 3. Installer "Mode" vs Forum "Mode"

phpBB has a clear two-mode concept:

| Aspect | Forum mode | Installer mode |
|---|---|---|
| Entry | `web/app.php` / `web/index.php` → `common.php` | `web/install.php` → `src/phpbb/install/app.php` |
| Guard constant | `PHPBB_INSTALLED` | `IN_INSTALL` |
| DI environment | `production` / `development` | `installer` |
| Extensions loaded | Yes | No |
| DB available | Yes (required) | No (DB may not exist yet) |
| Config PHP loaded | Yes | No |
| Container | `$phpbb_container` | `$phpbb_installer_container` |

There is **no hybrid mode** — the two are mutually exclusive entry paths.

---

## 4. What Environment `web/install/api.php` Needs

Because the install REST API will be called:
- **Before** phpBB is installed (e.g. `POST /install/api/install`) — DB may not exist, `config.php` is absent or incomplete
- **During** the install process — DB may be partially set up
- **Possibly after** install — not needed; at that point regular `app.php` handles everything

Therefore, `web/install/api.php` **cannot safely include `common.php`**:
- If `PHPBB_INSTALLED` is not defined, execution redirects to the HTML installer — a 302 response instead of JSON
- Even if `PHPBB_INSTALLED` were faked, `common.php` would try to build the forum container, which calls `dbal.conn`, which needs live DB credentials

---

## 5. Option Analysis & Recommendation

### Option A: Skip `common.php`, build own minimal boot

Implement `web/install/api.php` the same way `web/install.php` does:

```php
define('IN_INSTALL', true);
define('PHPBB_ROOT_PATH', __DIR__ . '/../../');
// Optionally define a REST-API variant to distinguish from HTML installer:
define('IN_INSTALL_REST_API', true);

require PHPBB_ROOT_PATH . 'src/phpbb/install/startup.php';

// $phpbb_installer_container is now available
$kernel = $phpbb_installer_container->get('http_kernel');
$request = $phpbb_installer_container->get('symfony_request');
$response = $kernel->handle($request);
$response->send();
$kernel->terminate($request, $response);
```

**Pros**: Follows phpBB's own pattern exactly; installer container is already designed for pre-install state; no DB required; no redirect guard issue.  
**Cons**: Must register REST controllers inside the installer routing (custom `installer_resources_locator` or additional routing YAML).

### Option B: Use installer-specific DI container (same as A in practice)

This is merely the same as A — "installer DI container" *is* the installer-specific DI container. No separate concept exists.

### Option C: Define `PHPBB_INSTALLED` before including `common.php`

```php
define('PHPBB_INSTALLED', true); // lie to bypass redirect
require __DIR__ . '/../../src/phpbb/common/common.php';
```

**Fatal flaw**: `common.php` then builds the full forum DI container including `dbal.conn`. If phpBB is not yet installed, `config.php` has no real DB credentials. The container build will fail with a connection error or an unresolvable service exception.

**Not viable** pre-install.

---

## 6. Recommendation

**Use Option A — mirror the `web/install.php` bootstrap exactly.**

Steps for `web/install/api.php`:
1. `define('IN_INSTALL', true)` — required for `startup.php` guard
2. `define('PHPBB_ROOT_PATH', ...)` — installer root
3. `require PHPBB_ROOT_PATH . 'src/phpbb/install/startup.php'` — builds `$phpbb_installer_container`
4. Register REST API controllers as Symfony controller services inside the installer DI config (a new `services_install_rest_api.yml` imported into `installer/container/services.yml`)
5. Register routes in the installer router (add a new routing YAML referenced by `installer_resources_locator`)

This approach:
- Avoids `PHPBB_INSTALLED` redirect entirely — `common.php` is never included
- Works before DB exists (installer container has no DB services)
- Shares the installer container with any install tasks that the REST API triggers
- Is idiomatic to phpBB's own architecture

> **Note**: After installation completes, the REST API for forum operations would use a **different** entry point (`web/app.php` / `web/api.php`) that includes `common.php` and builds the full forum container. The two entry points serve different lifecycle phases.
